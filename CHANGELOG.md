## [Unreleased]

### Removed

- **`_legacy_reference/` gelöscht.** Verzeichnis enthielt die ursprüngliche `Ynamite\Massif\Media`-Quelle aus `redaxo-massif`, gehalten für Porting-Referenz beim Greenfield-Skeleton. Addon ist jetzt produktiv verifiziert; Referenzkopie ist obsolet (Original-Historie liegt in `redaxo-massif`). Connect-Installs sparen ein paar KB; CLAUDE.md-Migrationsabsatz entsprechend gestrichen. Begleitend in derselben Änderung: `CLAUDE.md`-Gotcha zum Glide-Crop-Decimal-Trap zeigt jetzt korrekt auf `FitTokenBuilder::parseFocalToInts()` (vorher fälschlich auf nicht-existierendes `PictureRenderer::parseFocalToInts()`).

- **Blurhash-Feature komplett entfernt.** `Image::blurhash($src)`, der `withBlurhashAttr()`-Builder-Step, das `data-blurhash`-Attribut, das `$blurhash`-Feld auf `ResolvedImage`, der Compute-Pfad in `MetadataReader` (inkl. `kornrunner/blurhash`-Dependency und vendor-Verzeichnis), das Blurhash-Fieldset im **Placeholder**-Settings-Tab, der Blurhash-Eintrag in der Meta-Cache-TTL-Notice (`pages/settings.security.php`), die `massif_media_blurhash_legend`-Sprachzeile und die zugehörigen Tests sind weg. Bestehende `_meta/`-Sidecars enthalten noch einen unbenutzten `blurhash`-Key — harmlos, beim nächsten Schreiben fällt er weg. Stale `rex_config`-Zeilen (`blurhash_enabled`, `blurhash_components_x`, `blurhash_components_y`) bleiben liegen — kein Reader, kein Effekt. Der bisherige `composer update`-Schritt zog auch eine transitive `intervention/image` 3.11.7 → 3.11.8 mit.

### Changed

- **Render-State-Resolution zentralisiert in `Pipeline\RenderContext`.** `PictureRenderer::render()` und `Preloader::drain()` haben vorher die gleiche Ableitungskette (effectiveRatio, effectiveFit, fitToken, effectiveMaxWidth, widths-Pool) jeweils inline implementiert; jeder Drift hätte stille URL-Divergenz produziert. Beide Pfade rufen jetzt `RenderContext::build(...)` einmal pro Entry auf und delegieren die srcset-String-Bildung an `RenderContext::buildSrcset(UrlBuilder, ResolvedImage, format, quality, filterParams)`. Pure Refactor für `PictureRenderer` (kein Verhaltenswechsel — alle bestehenden Tests bleiben grün); Bugfix-Refactor für `Preloader` (siehe Fixed-Eintrag unten). `RenderContext` ist eine readonly Value Object analog zu `ResolvedImage`; die Konstante `RATIO_EQUAL_EPSILON` wandert von `PictureRenderer` zu `RenderContext`. Tests in `tests/Unit/Pipeline/RenderContextTest.php` (11 Tests / 28 Assertions) decken die Resolution-Regeln plus die srcset-Helper ab.

- **`color_enabled` ist auf neuen Installationen jetzt standardmäßig an** (`Config::DEFAULTS[KEY_COLOR_ENABLED] = 1`, vorher `0`). Dominante Farbe + LQIP zusammen sind die optisch beste Default-Kombi: die Farbe paintet sofort, das LQIP überlagert sobald dekodiert, die fertige Variante überschreibt beides. Bestehende Installationen sind nicht betroffen — sobald der Schlüssel einmal in `rex_config` geschrieben wurde (auch als unticked → `null`), respektiert `Config::checkboxBool` via `array_key_exists` den User-Wert und ignoriert das Default. Nur Installationen, die das Setting nie angefasst haben, sehen den geänderten Default. Imagick wird benötigt; ohne sie returnt der Encoder still `''` und die Default-on-Konfiguration wirkt sich nicht aus.

- **LQIP-Default `blur` auf `5` (vorher `40`).** Liefert eine merklich softere Inline-Vorschau. Installs, die den Wert nie über die Settings-Seite gesetzt haben, sehen automatisch das neue Verhalten — bestehende `_lqip/`-Cache-Dateien sind aber nach altem Wert generiert; einmal `Cache leeren`, damit sie neu berechnet werden.
- **LQIP-Format auf WebP umgestellt** (vorher JPEG); Inline-Data-URI ist jetzt `data:image/webp;base64,…`. Bei gleichem Blur / Qualität deutlich kleiner als JPEG für die LQIP-Größe. WebP-Browser-Support ist seit Safari 14 universal.
- **EXIF / XMP / IPTC / ICC-Profil werden vor dem Encoden gestrippt — für jede generierte Variante.** Neuer `lib/Glide/StripMetadata.php` Manipulator (Imagick-only, ruft `stripImage()`), läuft unconditionally nach `ColorProfile`. Bandbreite: für ein iPhone-Asset ~10–30 KB pro Variante × 3 Formate × 5–10 Breiten Ersparnis. Privacy: GPS-Koordinaten, Face-Detection-Bounding-Boxes, Photographer-Credit, Depth-Maps, Maker-Notes verlassen den Server nicht mehr. Korrektheit: `ColorProfile` normalisiert Pixel bereits zu sRGB via `transformImageColorspace()`, lässt aber das eingebettete ICC-Profil unangefasst — eine Display-P3-Quelle endete damit mit sRGB-Pixeln, die ein P3-Profil claimen, was color-managed Browser misrendern. Strip behebt diesen latenten Bug nebenbei. **Bestehende Cache-Varianten auf Disk tragen weiterhin Metadaten** — einmal `Cache leeren` nach Update, damit sie gestrippt neu generiert werden. (Anfangs ge-shipped als LQIP-only über ein `Server::$activeStripMetadata`-Flag — eine Iteration später für alle Varianten geöffnet, sobald klar war, dass die ColorProfile-Inkonsistenz ohnehin Profile-Strip rechtfertigt.)
- **`Pipeline\Placeholder::CACHE_VERSION = 'v2'`** hängt sich in den `_lqip/`-Cache-Key (`xxh64({src}:{mtime}:{version})`) — bestehende `_lqip/*.txt` invalidieren sich beim nächsten Read automatisch (würden sonst stale jpg-encodierte Data-URIs zurückgeben). Kein manueller `Cache leeren`-Schritt nötig für die Format- / Strip-Änderung.

### Fixed

- **Preload-`<link>`-URLs jetzt bit-identisch mit den `<img srcset>`-URLs.** Zwei latente Divergenzen zwischen `Preloader::drain()` und `PictureRenderer::render()` haben dazu geführt, dass der Browser eine Preload-Variante nicht mit dem späteren `<img>`-Fetch entduplizieren konnte: (a) `Preloader` baute ein `cover-X-Y`-Fit-Token auch dann, wenn das angefragte Ratio innerhalb der `RATIO_EQUAL_EPSILON`-Toleranz vom intrinsischen Ratio lag — `PictureRenderer` lässt das Token in dem Fall weg. Konkretes Symptom: `ratio="16:9"` auf einer 1600×900-Quelle → `<img srcset>` zeigte `cache/hero.jpg/avif-1200-q.avif`, das Preload-`<link imagesrcset>` zeigte `…/avif-1200-675-cover-50-50-q.avif`; zwei verschiedene Cache-Pfade für dasselbe Bild, der Preload-Fetch warmt nicht den URL, den der Browser danach für das `<img>` auflöst. (b) `Preloader` wendete den `effectiveMaxWidth`-Cap nicht an, den `PictureRenderer` für `COVER`/`CONTAIN` benutzt (z. B. 4000-Cap bei 1:1-Crop auf 5000×4000) — Preload-URLs oberhalb des Caps existierten nirgends im gerenderten `<img srcset>`. Beide Bugs waren Konsequenz duplizierter Resolution-Logik. Fix über die Adoption von `Pipeline\RenderContext` (siehe Changed-Eintrag): beide Pfade laufen jetzt durch `RenderContext::build()` und teilen die ratio-equal-epsilon Short-Circuit- und Width-Cap-Entscheidungen — URLs string-matchen by construction. Zwei neue Tests in `tests/Unit/Pipeline/PreloaderTest.php` (`testDrainOmitsCropTokenWhenRatioMatchesIntrinsic`, `testDrainCapsWidthsAtEffectiveMaxForCoverCrop`) lasen das fixierte Verhalten ein.

- **Cache-Invalidierung beim Update einer Mediapool-Datei (Focus-Point-Bug).** Bisher: User ändert den Focus-Point eines Assets via `focuspoint`-Addon-UI, klickt Speichern — Frontend zeigt weiter den alten Crop. Ursache: das `focuspoint`-Addon schreibt nur die DB-Spalte `med_focuspoint`, ohne die Datei auf Disk anzufassen. Der `mtime` bleibt stabil, der Meta-Cache-Hash (`xxh64(filename:mtime)`) entsprechend auch — `meta.json` liefert den **alten** Focus-Point zurück, der URL-Builder backt den alten `cover-X-Y`-Token in das gerenderte `<picture>`-Markup, der Browser bekommt denselben Cache-Hit wie vorher. Ohne aktive Invalidierung war ein manuelles "Cache leeren" der einzige Weg — der den Cache aller anderen Assets gleich mit nukte. Fix: neue `lib/Pipeline/CacheInvalidator.php` löscht alle vier Cache-Buckets (`cache/{filename}/` Varianten-Verzeichnis, `_meta/`-Sidecar, `_lqip/`-Data-URI, `_color/`-Hex) für eine einzelne Datei in O(1). `boot.php` registriert sich auf REDAXOs `MEDIA_UPDATED` und `MEDIA_DELETED` Extension-Points (beide tragen `params['filename']` mit demselben Shape — eine geteilte Closure macht den Job). Begleit-Refactor: `MetadataReader::metaCachePath` ist jetzt `static` (war Instance-Method), `Placeholder` und `DominantColor` exposen je eine `public static cachePathFor(string $filename, int $mtime)` für die Invalidator-Lookups — keine Duplikation der Cache-Key-Formel. Datei-Replacement-Updates haben einen kleinen Orphan-Effekt (alte Hash-gekeyede Sidecars bleiben unter dem alten `mtime`-Hash liegen, ~50 B – 2 KB pro Datei) — bewusst akzeptiert, das Varianten-Verzeichnis (der bulky Teil) wird immer komplett gelöscht. Tests: `tests/Unit/Pipeline/CacheInvalidatorTest.php` lockt alle vier Buckets, Sibling-Isolation, gelöschte-Source-Datei-Pfad, leerer-Filename-Defensive und den Datei-Replacement-Orphan-Tradeoff ein; `tests/Integration/CacheInvalidatorTest.php` exerziert den End-to-End-Flow gegen `MetadataReader` mit echtem Fixture (Read → Cache vorhanden → invalidate → Cache weg → Read → Cache neu da). README-Abschnitt **Cache-Invalidierung** entsprechend erweitert.

- **`<link rel="preload">` bekommt jetzt `fetchpriority="high"`.** Lighthouses **LCP request discovery**-Audit hatte zwei Häkchen: "Request is discoverable in initial document" (vorher schon erfüllt durch den Preload selbst) und "fetchpriority=high should be applied" (vorher fehlend). Ohne das Attribut sitzt der Preload-Fetch auf der Default-Image-Priority und konkurriert mit Below-Fold-Lazy-Bildern auf langsamen Verbindungen — der Audit weist explizit darauf hin, dass Preload allein nicht reicht. Da `preload(true)` im Addon semantisch "Above-the-Fold-Hero / LCP-Kandidat" bedeutet, ist `fetchpriority="high"` immer-an korrekt; kein Opt-out, kein neues Setting. Test in `tests/Unit/Pipeline/PreloaderTest.php`.

- **Checkbox-Settings ließen sich aus dem Backend nicht **deaktivieren** (LQIP, CDN, Dominante Farbe).** Folge-Bug zum vorherigen Aktivieren-Fix: das damalige `Config::checkboxBool` ging davon aus, ein un-getickter Save persistiert als `''`. Tatsächlich submitten Browser unchecked Checkboxen gar nicht, `rex_form_base::elementPostValue` returnt `null`, `config_form::save` ruft `rex_config::set(ns, key, null)` — der Wert landet als literal `null` in `rex_config`. Die alte Reader-Logik (`if ($raw === null) $raw = DEFAULTS[$key]`) konnte "User hat aktiv un-getickt" nicht von "Schlüssel nie geschrieben" unterscheiden, weil `rex_config::get` beide Fälle mit `??` zu `null` collapsed UND `rex_config::has` `isset`-basiert ist (returnt false für null-Werte). Symptom: User deaktiviert LQIP via Checkbox, speichert, aber `<img style="background-image:…">` taucht weiterhin auf, weil `lqipEnabled()` auf den DEFAULT 1 zurückfällt. Fix: `checkboxBool` liest jetzt das ganze Namespace-Dict via `rex_config::get(ADDON)` und prüft mit `array_key_exists` — das ist der einzige öffentliche Weg, "explizit null" von "nie geschrieben" zu unterscheiden. Explizit-null-Werte produzieren jetzt `false`. Test in `tests/Unit/ConfigTest.php` lockt den null-as-false Pfad und die nie-geschrieben-vs-explizit-null Distinktion ein. Test-Stub `rex_config::get` um den No-Key-Aufruf erweitert (gibt das ganze Namespace-Dict zurück, matched REDAXOs core API). CLAUDE.md-Gotcha entsprechend umgeschrieben — vorheriger Wortlaut implizierte fälschlicherweise `''` als unticked-Shape, was nie zutraf.

- **Checkbox-Settings (LQIP, Blurhash, CDN) ließen sich nicht von der UI aktivieren.** `rex_config_form::addCheckboxField` speichert getickte Checkboxen pipe-delimited (`'|1|'`), weil eine Checkbox-Gruppe mehrere Optionen halten kann. Die alten Reader (`Config::lqipEnabled` / `blurhashEnabled` / `cdnEnabled`) machten `(bool) (int) self::get(...)` — PHP int-castet `'|1|'` zu `0`, was die User-Aktivierung still wieder auf "aus" geflippt hat. Symptom: Box gehakt, gespeichert, kein LQIP / Blurhash / CDN-Output. Fix: neuer `Config::checkboxBool()` Helper geht direkt an `rex_config::get` (bypasst die Default-Fallback-Logik von `Config::get`, die einen User-Untick als "fehlend" behandelt und stattdessen das Default zurückgibt — wodurch sich eine Default-on Setting nicht abschalten ließe), strippt Pipes, dann int-castet. Drei Reader umgestellt. Tests in `tests/Unit/ConfigTest.php` locken die drei Storage-Shapes (`'|1|'`, `''`, plain int) ein. Gotcha in CLAUDE.md dokumentiert, damit künftige Checkbox-Settings nicht in dieselbe Falle treten.

### Added

- **GitHub-Actions-Workflow für automatisches Release-Publishing** (`.github/workflows/publish-to-redaxo.yml`). Triggert auf `release: published`: setzt PHP 8.2 + Composer auf, installiert Dependencies inkl. Dev (für Tests), führt `composer test:unit` als Quality-Gate aus, reinstalliert ohne Dev (`composer install --no-dev`), verifiziert via `bin/check-vendor`, dass keine PHPUnit-Family-Pakete im `vendor/` übrig sind, baut `massif_media.zip` mit massif-spezifischer Exclude-Liste (`.git/`, `.github/`, `.claude/`, `.worktrees/`, `bin/`, `tests/`, `phpunit.xml`, `.phpunit.cache/`, `docs/superpowers/`, `*.DS_Store`, `*Thumbs.db`, `*.log`), hängt das Zip an die GitHub-Release an (`softprops/action-gh-release@v2`) und veröffentlicht über `FriendsOfREDAXO/installer-action@1.2.0` auf my.redaxo.org. Voraussetzungen: einmalige Addon-Registrierung auf my.redaxo.org und die Repo-Secrets `MYREDAXO_USERNAME` / `MYREDAXO_API_KEY`. Pattern adaptiert vom Schwester-Addon `viterex-addon`; strukturelle Unterschiede: keine Node/JS-Build-Stufe (massif_media hat keine), Unit-Test-Gate vor dem Zip (viterex hatte keinen), Exclude-Liste an die tatsächliche Repo-Layout angepasst. Integration-Tests laufen bewusst nicht im CI — sie bräuchten Imagick + Temp-FS auf dem Runner; die Unit-Suite ist der pragmatische Gate. Manuelle Release-Schritte zukünftig: `version:` in `package.yml` bumpen, CHANGELOG-Eintrag, Commit, `gh release create vX.Y.Z` — den Rest macht der Workflow.

- **Cache-Statistik-Panel auf dem Sicherheit & Cache-Tab.** Neue `lib/Pipeline/CacheStats.php` läuft rekursiv über `assets/addons/massif_media/cache/`, summiert Bytes + Datei-Anzahl pro Kategorie (`meta` / `lqip` / `color` / `animated` / `variants`) und merkt sich Älteste/Neueste Datei-`mtime`. Output rendert als Tabelle im bestehenden **Cache**-Panel über dem "Cache leeren"-Button. Memoization in `cache/_stats.json` mit 5-Min-TTL hält wiederholte Page-Loads billig; ein neuer "Statistik neu berechnen"-Button bypasst den Memo on-demand. `_stats.json` wird beim "Cache leeren" durch `rex_dir::delete($cacheDir, false)` automatisch mit weggeräumt — frische Stats beim nächsten Visit. Read-only: keine Per-Variant-Löschung, keine Eviction-Policy. Tests in `tests/Unit/Pipeline/CacheStatsTest.php` decken leeren Cache, fehlendes Verzeichnis, Kategorisierungs-Regeln (alle 5 Kinds), Memo-Hit innerhalb TTL, `forceRefresh`-Bypass und Oldest/Newest-Tracking ab.

- **Animierte GIFs werden automatisch als animiertes WebP angeboten.** Neue `lib/Pipeline/AnimatedWebpEncoder.php` re-encodiert animierte GIF-Quellen in eine einzelne animierte WebP-Variante (intrinsische Dimensionen, kein Multi-Width-Pool). Glides Standard-Encoder ist single-frame und droppt alle Frames nach dem ersten still — der neue Encoder bypasst Glide und nutzt direkt `Imagick::coalesceImages() + writeImages($path, true)`. `MetadataReader` bekommt eine `probeAnimated()`-Methode (Imagick `pingImage` + `getNumberImages > 1`, nur für Formate die Animation tragen können — gif/webp/png), persistiert das Ergebnis im `_meta/`-Sidecar als `is_animated`. Bestehende Cache-Files ohne den Key werden als `false` gelesen — keine Migration nötig. `ResolvedImage` bekommt `readonly bool $isAnimated = false`. `PassthroughRenderer` (für SVG/GIF-Quellen) wickelt animierte GIFs in `<picture><source type="image/webp" srcset="...animated.webp"><img src="...gif"></picture>` ein; nicht-animierte Quellen, SVGs und CDN-Modus fallen auf das bisherige `<img>`-Markup zurück. URL-Format: `cache/{src}/animated.webp` (single segment), HMAC-signiert über den bestehenden `Signature::sign`-Pfad ohne Filter-Blob. `Endpoint::handle` matcht das `/animated.webp`-Suffix vor `parseCachePath` und dispatcht zum Encoder; Glide wird nicht angefasst. Tests: `tests/Unit/Pipeline/AnimatedWebpEncoderTest.php` für Gates (statisch, non-GIF, cache hit), `tests/Integration/AnimatedWebpEncoderTest.php` mit echtem 3-Frame-Fixture (verifiziert Multi-Frame-Output und Cache-Idempotenz), `tests/Unit/View/PassthroughRendererTest.php` um Wrap- und Fallback-Pfade erweitert, `MetadataReaderTest` lockt `is_animated`-Detection für animierte GIF / statisches GIF / JPEG ein. Neuer Fixture: `tests/_fixtures/animated-3frame.gif` (3 Solid-Color-Frames). **CLAUDE.md-Gotcha festgehalten**: Glide-Encoder ist single-frame; jeder animierte Output braucht den `coalesceImages → writeImages(adjoin=true)`-Pfad und muss an Glide vorbei. README-Abschnitt **Animierte GIFs** ergänzt.

- **Dominante Farbe als optionaler Placeholder.** Neue `lib/Pipeline/DominantColor.php` extrahiert pro Bild eine repräsentative Hex-Farbe via Imagick `quantizeImage(1, COLORSPACE_SRGB)` auf einer 50 px breiten Working-Copy (~5–20ms typisch). Wird als `background-color` ins bestehende `style`-Attribut des `<img>` geschrieben — vor `background-image` (LQIP) und `object-position` (Focal), sodass die Farbe sofort paintet und die LQIP-Image-URI sich darüberlegt sobald dekodiert. **Unabhängig** von `lqip_enabled`: Beide können gleichzeitig an sein (Farbe sofort + LQIP sobald geladen + finale Variante überschreibt beides) oder nur einer von beiden — der User entscheidet pro Setting. Default ist deaktiviert (`color_enabled = 0`). Cache-Layout spiegelt die LQIP-Konvention: `cache/_color/<2-char-prefix>/<xxh64>.txt` mit `CACHE_VERSION`-Constant für Self-Invalidation. `PictureRenderer`-Konstruktor bekommt `DominantColor` als vierten DI-Param, `ImageBuilder::render()` instanziiert ihn. Tests: `tests/Unit/Pipeline/DominantColorTest.php` deckt die Gate-Bedingungen ab (disabled, passthrough, cache hit, unreadable source); `tests/Integration/DominantColorTest.php` exerziert den echten Imagick-Pfad gegen Fixture-Bilder mit Toleranz ±8 pro Kanal; `tests/Unit/View/PictureRendererTest.php` lockt die Style-Reihenfolge (Color-vor-LQIP-vor-Focal) und Color+Focal-Combo ein. Settings-UI: neues Fieldset auf dem **Placeholder**-Tab. **Imagick-Gotcha festgehalten in CLAUDE.md**: `COLORSPACE_RGB` (Imagick-Speak für linear-light) bei Quantisierung produziert gamma-falsche, viel zu dunkle Farben; `COLORSPACE_SRGB` ist Pflicht für alles, was der User danach sieht. Bei Q16-Builds zusätzlich `getColor(2)` für 0-255-Channel-Werte (sonst 0-65535). README-Abschnitt **Dominante Farbe** ergänzt; `color_enabled` in der Konfigurations-Tabelle.

- **`rex::isDebug()`-gated Missing-Src-Marker für `Image`- und `VideoBuilder`.** Bisher: ein typo'tes `src=` im REX_PIC/REX_VIDEO oder ein Asset, das aus dem Mediapool entfernt wurde, produzierte einfach leeres HTML — keine Hinweis im DevTools, kein Signal für den Editor, manchmal kein Logfile-Eintrag (Video-Pfad war komplett still). Jetzt: `VideoBuilder::render` loggt den unlesbaren Pfad via `rex_logger` (Parität mit dem `ImageBuilder`-Catch); zusätzlich rendert ein HTML-Kommentar `<!-- massif_media: src not found "<filename>" -->` an die Stelle des Bildes/Videos, wenn `rex::isDebug()` an ist. Production (`debug: false`) bleibt unverändert leeres HTML — keine Sichtbarkeit nach außen, aber der Logfile-Eintrag erscheint trotzdem. Filename ist HTML-escaped (`<script>`-Injection im Mediapool-Namen wird neutralisiert). Tests in `tests/Unit/Builder/MissingSrcMarkerTest.php` decken Production / Debug / leeren Filename / Image- + Video-Pfad / Escaping ab. README-Troubleshooting-Abschnitt erweitert.

- **Unit-Tests für `View\PictureRenderer` und `View\PassthroughRenderer`** (`tests/Unit/View/PictureRendererTest.php`, `tests/Unit/View/PassthroughRendererTest.php`). Vorher hatten die beiden View-Renderer keine direkten Unit-Tests — Coverage kam nur transitiv über die Integration-Tests, die langsam und beim Debuggen unhandlich sind. Neue Tests decken die Renderer-Entscheidungen isoliert ab: `effectiveRatio`-Ableitung (width+height vs. ratio-explicit vs. intrinsic), `needsCrop`-Fast-Path bei `RATIO_EQUAL_EPSILON`-Match, `effectiveMaxWidth`-Cap für COVER/CONTAIN aber Bypass für STRETCH, `Fit::NONE` skippt Crop komplett, Focal-Point-Default `50% 50%`, `object-position`-Style nur wenn Focal gesetzt, LQIP-Cache-File pre-seeding (umgeht Glide), LQIP+Focal kombiniert in einem Style-Attr, `aria-hidden="true"` bei null/empty `alt`, leerer Width-Pool bei `intrinsicWidth=0` returnt `''`, Custom-Quality-Override pro Format, Filter-Blob-`&f=`-Anhängen, HTML-Escaping in `alt`. 31 neue Tests / 61 Assertions, Suite läuft weiterhin unter 50ms. Begleitend: `tests/_stubs/redaxo.php` bekommt `rex_url::base()`, das vom `PassthroughRenderer` benötigt wird.

- **Settings-Hilfetexte (`setNotice`) für alle bisher kommentarlosen Felder** — die drei Format-Qualitäts-Inputs auf dem **Allgemein**-Tab (`Config::KEY_QUALITY_AVIF/WEBP/JPG`) und die drei LQIP-Knöpfe auf dem **Placeholder**-Tab (`Config::KEY_LQIP_WIDTH/BLUR/QUALITY`) bekommen je einen kurzen deutschen Erklär-Satz mit Default-Hinweis. Lücken-Schließung gegenüber den restlichen Feldern, die schon vorher `setNotice` hatten — keine API- oder Verhaltensänderung. Wording lehnt sich an den README-LQIP-Abschnitt und die Defaults-Tabelle an, damit Terminologie konsistent bleibt.

- **`MetadataReader::computeBlurhash`** auseinandergeschnitten in drei Zuständigkeiten: `downscaleIfOversized()` (resize via Glide-style 64-px Cap), `extractPixels()` (RGB-Pixel-Matrix-Extraktion via `imagecolorat`), und der Top-Level-Flow (load → downscale → extract → encode). Vorher 51 Zeilen prozedural mit drei Verantwortlichkeiten in einem Methodenkörper; jetzt eine 5-Zeilen-Sequenz. Kein Verhaltenswechsel — Hash-Output bit-identisch zum Vorzustand.
- **`Glide\CacheKeyBuilder`** zentralisiert die Filter-Param-Hash- und Filter-Blob-Erzeugung. Vorher in fünf Stellen dupliziert: dreimal die `ksort + json_encode + md5 + substr 8` Hash-Recipe (in `lib/Pipeline/UrlBuilder.php`-`build`, `lib/Glide/Endpoint.php`-`handle`, `lib/Glide/Server.php`-`cachePath`) und dreimal die `base64UrlEncode` (zwei Mal in `UrlBuilder` als private Method, einmal in `tests/Integration/FilterPipelineTest.php`). Die Klasse exponiert `hashFilterParams()` (8 hex chars) und `encodeFilterBlob()` / `decodeFilterBlob()` (base64url JSON); die geteilte ksort-Disziplin lebt in einer privaten `canonicalJson()` Methode. `UrlBuilder::base64UrlDecode` und `base64UrlEncode` entfallen — der einzige externe Konsument war `Endpoint`, der jetzt direkt an `CacheKeyBuilder::decodeFilterBlob` delegiert. Pure Refactoring; identisches On-Disk- und Wire-Verhalten (Hash-Algorithmus, Padding-Strip, ksort-Reihenfolge unverändert).
- **`Glide\FitTokenBuilder`** zentralisiert die `Fit` + Focal-Point → `cover-X-Y` / `contain` / `stretch` Übersetzung. Vorher in zwei Stellen identisch dupliziert (private `buildFitToken` in `lib/View/PictureRenderer.php` und `lib/Pipeline/Preloader.php`); `parseFocalToInts` wurde zusätzlich aus `PictureRenderer` cross-class von `Preloader` aufgerufen. Beide Aufrufer delegieren jetzt an die neue Klasse, der `PictureRenderer::parseFocalToInts`-Cross-Class-Aufruf in Preloader entfällt. Tests entsprechend verschoben (`tests/Unit/View/PictureRendererHelpersTest.php` → `tests/Unit/Glide/FitTokenBuilderTest.php`) und um `build()`-Coverage erweitert. Pure Refactoring — kein Verhalten ändert sich, kein API-Bruch (die alten privaten Methoden hatten keine externen Konsumenten).
- **Blurhash-Komponenten konfigurierbar** — neue Settings-Felder `Komponenten X` / `Komponenten Y` im Placeholder-Tab (`pages/settings.placeholder.php`). Vorher hardgecodet `Blurhash::encode($pixels, 4, 3)` in `MetadataReader::computeBlurhash`; jetzt aus `Config::blurhashComponentsX()` / `blurhashComponentsY()`. Werte werden auf `[1, 9]` geclampt (kornrunner-Constraint). Defaults `4` / `3` matchen das bisherige Verhalten — Migration ist no-op. Höhere Werte = mehr visueller Detail im decodierten Vorschaubild + längerer Hash-String. Änderungen werden erst nach `Cache leeren` wirksam, da bestehende `_meta/`-Sidecars den alten Hash gecached haben.
- **`REX_VIDEO[…]` Placeholder** — neues `rex_var` analog zu `REX_PIC`, gibt `<video>`-Markup aus (`lib/Var/RexVideo.php`). Exponiert die volle `Video::render()`-Argument-Surface: `src` (Pflicht), `poster`, `width`, `height`, `alt`, `class`, `preload`, `loading` als String/Int-Passthroughs sowie `autoplay`, `muted`, `loop`, `controls`, `playsinline` als Bool-Attribute (akzeptiert `"true"` / `"false"` / `"1"` / `"0"` / `"yes"` / `"no"`). Bool-Attribute werden _nur_ emittiert, wenn der Editor sie explizit setzt — die asymmetrischen Defaults aus `Video::render()` (`autoplay`/`muted`/`loop` default false, `controls`/`playsinline` default true) bleiben damit erhalten. Substitution beim Article-Cache-Rebuild (`Cache leeren` nach Update nötig, damit bestehende Slices neu gebaut werden). Unit-Tests in `tests/Unit/Var/RexVideoTest.php`.
- **Glide filter passthrough** — kompletter Manipulator-Surface über `Image::picture(filters: [...])`, `Image::for(...)->brightness/contrast/gamma/sharpen/blur/pixelate/filter/bg/border/flip/orient/watermark(...)` und 19 neue REX_PIC-Attribute (`brightness`, `contrast`, `gamma`, `sharpen`, `blur`, `pixelate`, `filter`, `bg`, `border`, `flip`, `orient`, `mark`, `marks`, `markw`, `markh`, `markpos`, `markpad`, `markalpha`, `markfit`). Numerische Ranges werden automatisch geclampt; ungültige Hex-Werte werden gedroppt; unbekannte Enum-Werte gehen an Glide durch (Glide ignoriert nicht-erkannte Filter still). Single-source Validation lebt in der neuen `lib/Glide/FilterParams.php`.
- **`assets/.gitignore`** — Cache-Dateien werden nie committet. Eine Ebene über `cache/` platziert (statt `assets/cache/.gitignore`), damit `rex_dir::delete($cacheDir, false)` (CACHE_DELETED-Hook + manuelles „Cache leeren") sie nicht versehentlich mitwegräumt. Inhalt: `cache/` — der Cache-Ordner wird komplett ignoriert; `install.php` legt ihn zur Laufzeit über `rex_dir::create()` an.
- **`install.php` schreibt das Runtime-`.gitignore` programmatisch** an `rex_path::addonAssets(Config::ADDON, '.gitignore')` mit Inhalt `cache/`. REDAXOs `installAssets()` verwendet `rex_finder` mit `ignoreSystemStuff=true`, dessen Filter alle Dateien droppt, die mit `.git` beginnen — gedacht für `.git/`-Verzeichnisse, fängt aber auch `.gitignore` mit ein. Die im Repo unter `assets/.gitignore` mitgelieferte Datei erreicht den Laufzeit-Pfad daher nie. Workaround: nach `installAssets()` den File-Content selbst schreiben (`rex_file::put`).
- **PHPUnit ^11 test suite** unter `tests/`. Zwei Suites: `tests/Unit/` (pure-logic, kein REDAXO / kein Glide) und `tests/Integration/` (echte Glide-Pipeline gegen Temp-FS mit Fixture-Bildern). Composer-Scripts `composer test`, `composer test:unit`, `composer test:integration`. REDAXO-Klassen werden in `tests/_stubs/redaxo.php` minimal gestubt (rex_config / rex_path / rex_url / rex_logger / rex_media / rex_dir / rex_file / rex / rex_var) — Stubs registrieren nur, wenn die echten Klassen nicht geladen sind. Fünf Fixture-Bilder unter `tests/_fixtures/`, regenerierbar via einmaligem `tests/_fixtures/generate.php`. ~50 Tests / ~140 Assertions, Unit-Suite läuft unter 5 Sekunden.
- **`bin/check-vendor`** — kleines Bash-Script, das `vendor/` nach bekannten PHPUnit-Family-Paketen scannt und mit Exit-Code 1 abbricht, wenn welche da sind. Schützt vor versehentlichem Committen von Dev-Paketen.
- **Cropping support via `fit` enum**: neue `lib/Enum/Fit.php` mit den Werten `cover`, `contain`, `stretch`, `none`. `Image::picture(fit: ...)`, `Image::for($src)->fit(...)`, `Pic::picture(fit: ...)` und `REX_PIC[... fit="..."]` akzeptieren das Argument. Default ist `cover` (fokuspunkt-bewusster Center-Crop), sobald `ratio` oder `height` gesetzt ist; ohne Ziel-Box wird das Argument still ignoriert. Fokuspunkt-Quelle: per-call `focal="..."` > Asset-`med_focuspoint` (focuspoint-Addon) > `50% 50%`. Glide-Param: `fit=crop-{focalX}-{focalY}` (Boundary-Translation aus unserem internen `cover-X-Y` Token). `contain` entspricht Glide's Standard-Behavior — proportionales Verkleinern bis das Bild in die Box passt, **kein Letterbox-Padding**. `stretch` ignoriert den `srcset`-Cap und kann auf jede Größe verzerren. `none` opt-out für das Verhalten vor diesem Release (nur Layout-Box, keine Bild-Manipulation).
- **Self-contained Cache-URL-Routing** via `PACKAGES_INCLUDED`-Hook (`lib/Glide/RequestHandler.php`, registered at `rex_extension::EARLY`). Cache-URLs der Form `/assets/addons/massif_media/cache/…`, die in REDAXOs Frontend `index.php` landen, werden vor `yrewrite` und Article-Rendering abgefangen, die Variante wird on-demand generiert und ausgeliefert, dann `exit`. Pattern aus REDAXOs eigenem `media_manager` adaptiert. Wirkung: das Addon funktioniert jetzt **ohne** addon-spezifische `.htaccess`-Rewrites, nginx-Snippets oder LocalValetDriver-Patches — auf Apache, standalone nginx, Laravel Herd und Valet.
- README-Abschnitt **Placeholder-Strategien: LQIP vs. Blurhash** mit Tabelle, die beide Strategien gegeneinander stellt (Speicherort, Größe in HTML, Decode-Pfad inkl. server-seitiger PHP-Decode-Möglichkeit über `kornrunner/blurhash`, visueller Charakter, Use-Cases). Antwort auf die Frage „sind die nicht redundant?" — sind sie nicht; LQIP ist der direkte Weg für Server-Rendered HTML, Blurhash spielt seine Stärke in JSON-APIs aus.
- nginx Support: `assets/nginx.conf.example` als optionaler Performance-Snippet für standalone nginx (per-Site `server`-Block mit eigenem `root`) — `try_files`-Fastpath für Cache-Hits, Long-lived `Cache-Control`. Funktioniert nur als Optimierung; das Addon liefert auch ohne diese Datei aus.
- Unit-Test-Coverage für `Pipeline\Placeholder::generate` (`tests/Unit/Pipeline/PlaceholderTest.php`): drei Cases — LQIP disabled liefert leeren String, LQIP disabled aber Blurhash enabled liefert weiterhin leer (LQIP- und Blurhash-Strategien sind unabhängig gegated), Passthrough-Quelle (svg/gif) liefert leer auch bei aktivem LQIP. Lock-in für die Gate-Bedingung in `Placeholder::generate` Zeile 23.

### Changed

- **Cache-Layout flippt zu asset-keyed**: `cache/{src}/{transform}.{ext}` statt `cache/{transform}/{src}.{ext}`. Alle Varianten einer Quelle leben in einem Verzeichnis — bessere Inspektion, einfacheres Per-Asset-Wipen. Source-Subdirectories (`gallery/2024/foo.jpg`) bleiben erhalten. **Bestehende Cache-Dateien aus alten Layout sind orphaned** — `Cache leeren` einmalig nach dem Update räumt sie auf.
- **Cache-Key erweitert um optionalen `f{hash}`-Suffix bei Filter-Anfragen**: `{fmt}-{w}-{q}-f{hash}.{ext}` (bzw. `{fmt}-{w}-{h}-{fitToken}-{q}-f{hash}.{ext}` mit Crop). `{hash}` = `substr(md5(json_encode(ksort(filterParams))), 0, 8)`. Hash ist deterministisch unabhängig von Filter-Reihenfolge (durch `ksort`).
- **URL gewinnt `&f=base64url(json)`-Query-Param** bei Filter-Anfragen. HMAC deckt jetzt `path|f` zusammen ab — Filter-Werte können nicht ohne Signatur-Bruch manipuliert werden. Anfragen ohne Filter haben kein `&f`, Signatur deckt nur den Pfad ab (bit-identisch zum Verhalten vor diesem Release).
- **`Signature::sign($path, ?string $extraPayload = null, ?string $key = null)`** und entsprechend `verify` akzeptieren einen optionalen extraPayload als zweites Argument. Production-Verhalten ohne extraPayload bleibt unverändert. Existing 3-arg Aufrufer (`verify($path, $sig, $key)`) müssen auf Named-Args (`key: $key`) umgestellt werden — der dritte Positional-Slot ist jetzt `$extraPayload`.
- **CDN-URL-Template gewinnt `{f}`-Token** für den Filter-Blob (leer wenn keine Filter). Existing Templates ohne `{f}` produzieren identische URLs wie zuvor.
- **`Server::setActiveFilters` / `clearActiveFilters`** — neuer statischer Test-Seam, gesetzt von `Endpoint::handle` vor `makeImage`-Aufrufen, damit Glide's interne `Server::cachePath`-Aufrufe (über `cachePathCallable`) den gleichen On-Disk-Pfad produzieren wie unsere URL-Emission.
- **`Signature::sign($path, ?string $key = null)` und `Signature::verify($path, $sig, ?string $key = null)`** akzeptieren einen optionalen injizierbaren Key. Production-Aufrufer übergeben weiterhin keinen Key — Default fällt auf `Config::signKey()` zurück (bit-identisches Verhalten). Unit-Tests können einen expliziten Key setzen, ohne REDAXO zu booten.
- **`Glide\Server::create(?string $sourceDir = null, ?string $cacheDir = null)`** akzeptiert optionale Source-/Cache-Directory-Overrides. Production-Aufrufer übergeben nichts — Default fällt auf `rex_path::media()` / `rex_path::addonAssets(Config::ADDON, 'cache/')` zurück. Integration-Tests können Glide auf ein Per-Test-Temp-Verzeichnis zeigen lassen.
- **Drei pure Focal-Parsing-Helper sind jetzt `public`** statt `private`: `MetadataReader::normalizeFocal`, `MetadataReader::formatFocal`, `PictureRenderer::parseFocalToInts`. Letzterer ist jetzt zusätzlich `static` (der Body referenziert kein `$this`). Unit-Tests können sie direkt aufrufen, ohne Reflection. Production-Verhalten unverändert.
- **`CLAUDE.md`-Regel „No tests" entfernt.** Ersetzt durch eine Testing-Sektion mit den neuen Composer-Scripts und einer Liste, was unit-getestet wird vs. was manuell auf Herd verifiziert bleibt (Frontend-Boot-Pfad, OUTPUT_FILTER-Preload-Injection, Backend-Documentation-Tab).
- **Cache-Key erweitert um optionale `h`/`fitToken` Segmente**: Cache-Pfade für gecroppte Varianten haben jetzt die Form `{fmt}-{w}-{h}-{fitToken}-{q}/{src}.{ext}` (z. B. `avif-800-800-cover-50-50-50/hero.jpg.avif`). Nicht-gecroppte Anfragen behalten die bisherige `{fmt}-{w}-{q}/...` Form bit-identisch — bestehende Cache-Dateien auf Disk bleiben gültig. `Endpoint::parseCachePath` versteht beide Formen; bei Crop-Anfragen wird `cover-X-Y` an der Glide-Boundary zu `crop-X-Y` übersetzt, und `Server::cachePath` normalisiert in der Gegenrichtung (`crop-X-Y` → `cover-X-Y`), damit der Cache-Pfad auf Disk konsistent zur emittierten URL ist und der Static-Direct-Fastpath bei Cache-Hits greift. CDN-URL-Templates haben zwei neue Tokens: `{h}` und `{fit}` (leer wenn nicht gecroppt).
- **REX_PIC migriert von OUTPUT_FILTER zu nativem `rex_var`**: neue `lib/Var/RexPic.php` extends `rex_var`, registered as `REX_PIC`. Substitution passiert beim Article-Cache-Rebuild (statt Regex auf jedem Page-Render gegen den fertigen HTML-Output). Konsequenzen: (1) Performance — kein Regex-Overhead pro Render, der Article-Cache ruft direkt `Image::picture(…)` auf. (2) Scope — `REX_PIC` greift jetzt nur in Slice-Content, nicht mehr in beliebigem Output. Damit ist auch Folgeproblem (3) gelöst: der Backend-Dokumentations-Tab zeigt `REX_PIC[…]` Code-Beispiele in Markdown-Codeblöcken jetzt als Code, weil `rex_var` auf Backend-Pages nicht feuert. Bestehende Slices mit `REX_PIC[…]` müssen einmal „Cache leeren" durchlaufen, damit der Article-Cache mit der neuen Variable neu gebaut wird.
- README **Webserver-Konfiguration** Section komplett umstrukturiert: vorher drei Subsections (Apache out-of-box / Standalone nginx / Laravel Herd Driver-Patch) mit schrittweisen How-Tos. Jetzt eine kompakte Tabelle, die festhält, dass das Addon überall self-contained läuft und die Snippets eine optionale Performance-Optimierung sind.
- README: Cache-Hit-Bullet und URL-Schema-Section beschreiben jetzt den `PACKAGES_INCLUDED`-Hook als kanonischen Cache-Miss-Path. `.htaccess` und nginx-Snippet sind nur noch für den Cache-Hit-Fastpath relevant.
- README: REX_PIC-Attribut-Tabelle um eine **Default**-Spalte erweitert. Jeder Wert (Loading `lazy`, Decoding `async`, FetchPriority `auto`, Sizes aus `default_sizes`, Focal aus dem `focuspoint`-Addon, Width/Height intrinsisch usw.) ist explizit dokumentiert, damit Redakteure auf einen Blick sehen, was bei Weglassen eines Attributs greift.
- Placeholder-Settings-Tab klargestellt: LQIP und Blurhash sind zwei unabhängige Strategien (LQIP = Inline-Base64-JPEG, JS-frei; Blurhash = kompakter Hash für Client-Side-Rendering oder API). Intro-Hinweis am Panel-Anfang, ausführlichere Notice-Texte unter beiden Toggles. Der Blurhash-Option-Label nennt jetzt explizit „Beim ersten Zugriff auf ein Bild berechnen und in der Asset-Metadata cachen".
- Placeholder-Settings-Tab: Blurhash-Notice und Intro nennen jetzt auch die **server-seitige** PHP-Decode-Option via `\kornrunner\Blurhash\Blurhash::decode($hash, $w, $h)` (gibt eine Pixel-Matrix zurück, aus der man JPEG/PNG enkodieren kann). Vorher war nur der client-seitige JS-Decoder erwähnt — passt jetzt zur README-Tabelle „LQIP vs. Blurhash". Die drei LQIP-Number-Felder (Breite, Blur, Qualität) tragen zusätzlich ein `(LQIP only)` Suffix im Label, damit der Scope auch ohne Fieldset-Header eindeutig ist.
- README: dedizierter Abschnitt **REX_PIC — Placeholder für Inhaltspflege** mit neun Beispielen (minimal, mit `sizes`, mit `ratio`, responsive, mit Preload, mit Focal-Point, mit CSS-Klasse, SVG-Pass-through, eingebettet in Markdown), vollständiger Attribut-Tabelle und Performance-Hinweis. Vorher nur einzeiliger Erwähnung.

### Removed

- `lib/Parser/REXPicParser.php` und der zugehörige `OUTPUT_FILTER`-Substitution-Pass für `REX_PIC` (ersetzt durch `lib/Var/RexPic.php`, siehe Changed). Der OUTPUT_FILTER-Hook bleibt für die `<link rel="preload">` Injection in den `<head>` bestehen — das ist ein anderes Problem, das nicht von Code-Block-Pollution betroffen ist.
- `assets/LocalValetDriver.snippet.php` (in Unreleased nie veröffentlicht): der `frontControllerPath()`-Interceptor war nur nötig, weil REDAXOs Frontend `index.php` in Herd ohne nginx-Rewrites nicht von der Cache-URL erreicht wurde. Mit dem `PACKAGES_INCLUDED`-Hook ist Herd jetzt nativ abgedeckt — Cache-Misses laufen über Valets default `frontControllerPath()` → REDAXO-Frontend → Hook fängt ab.
- `assets/herd.conf.snippet` (in Unreleased nie veröffentlicht): der Server-Level-Rewrite-Ansatz funktioniert in Herd nicht — Valets `server.php` ignoriert nginx-`rewrite`-Direktiven, weil sie `$request_uri` nicht anfassen.

### Fixed

- **`ratio` und `height` croppen jetzt tatsächlich das Bild** — bisher änderten sie nur die HTML-`width`/`height`-Attribute (Layout-Box), während das von Glide generierte Bild proportional auf die `width` skaliert blieb. Resultat: bei `ratio="1:1"` auf einem 16:9 Quellbild wurde die Layout-Box quadratisch, das Bild aber 16:9 — Browser hat ohne `object-fit: cover` ein verzerrtes Ergebnis gezeigt. Jetzt: setzt `ratio` oder `height` ein Ziel-Format, wird das Bild via Glide auf dieses Format gecroppt (Default `fit="cover"`, fokuspunkt-bewusst). Das alte Verhalten (Box reservieren, Bild nicht anfassen) bleibt opt-in via `fit="none"`. **Visueller Bruch für bestehende Slices, die auf das alte Verhalten gebaut haben** — nach dem Update einmal Cache leeren, betroffene Slices identifizieren und ggf. `fit="none"` hinzufügen.
- `srcset` wurde fälschlicherweise auf den vom Aufrufer übergebenen `width` Parameter gecappt — bei `width=720` und intrinsisch 5000 px endete das `srcset` bei 720 w. Konsequenz: HiDPI-Screens (2×, 3×) konnten keine schärfere Variante anfordern, obwohl die Quelle sie hergeben würde, und mussten das 1×-Bild unscharf hochskalieren. `SrcsetBuilder::build()` nimmt den `maxWidth` Parameter nicht mehr — gecappt wird ausschließlich an der intrinsischen Quellbreite, das `width` Attribut bleibt rein layout-relevant. Verhalten richtet sich jetzt nach `next/image`'s responsive Modus (sobald ein `sizes` Attribut vorhanden ist — und in diesem Addon ist es das immer, weil `Config::defaultSizes()` automatisch greift). Wer das `srcset` explizit eingrenzen will, nutzt `Image::for(...)->widths([...])`.
- Glide cache-path callable schlug nach dem static-closure-Fix mit „Call to undefined method `League\Glide\Server::cachePath()`" fehl, sobald `Image::picture()` einen ersten Cache-Miss verarbeitete. Glide ruft `Closure::bind($callable, $this, static::class)` und rebindet damit nicht nur `$this`, sondern auch `self::` / `static::` auf seine eigene Server-Klasse. Die Closure verwendete `self::cachePath()` und resolvte deshalb zur Laufzeit gegen `League\Glide\Server` statt gegen `Ynamite\Media\Glide\Server`. Behoben durch Klassennamen-Referenz (`Server::cachePath(…)`) — wird zur Compile-Zeit über die File-Namespace-Auflösung resolvt und ist von der bound scope der Closure unabhängig.
- `Image::picture()` löste „Cannot bind an instance to a static closure" in `vendor/league/glide/src/Server.php` aus. Glide ruft `Closure::bind($callable, $this, static::class)` auf der Cache-Path-Callable auf — das schlägt bei `static fn (…)` fehl, weil statische Closures kein `$this` zulassen. `static` aus `Glide\Server::cachePathCallable()` entfernt; der Body nutzt sowieso nur `self::cachePath()`, das Binden ist also faktisch ein No-Op.
- Number-Inputs auf den Settings-Tabs (AVIF/WebP/JPG-Qualität, LQIP-Maße, Cache-TTLs) fehlte die `form-control` CSS-Klasse — REDAXO's `addTextField` injiziert sie automatisch, `addInputField` jedoch nicht. Inputs werden jetzt konsistent mit den Text-Feldern gerendert.
- Breite der Number-Inputs auf 100 px (Qualität / LQIP) bzw. 140 px (TTLs) begrenzt — `form-control` setzt sonst 100 % Container-Breite, was bei 1–3-stelligen Werten unverhältnismäßig wirkt.
- Default-Werte werden auf den Number-Inputs (Qualität / LQIP / TTLs) als `placeholder` angezeigt, damit auf frischen Installationen ohne gespeicherten `rex_config`-Wert ersichtlich ist, was bei leerem Feld als Default greift. Quelle ist `Config::DEFAULTS`, damit Code- und UI-Default nicht auseinanderlaufen.

## [1.0.0] — 2026-05-01

Erstes Release. MASSIF Media wurde aus dem `redaxo-massif` Sammel-Addon ausgelagert und auf eine moderne, eigenständige Pipeline umgebaut. Alte `Ynamite\Massif\Media\…` Aufrufe in bestehenden Projekten bleiben unverändert nutzbar (das alte Addon koexistiert); neuer Code verwendet `Ynamite\Media\…`.

### Added

- **`<picture>` Markup** mit AVIF, WebP und JPG Sources. Browser-seitige Format-Negotiation via `<source type="…">` — kein Accept-Header-Sniffing mehr.
- **On-demand Resizing** über [`league/glide`](https://glide.thephpleague.com/) (Imagick-Driver bevorzugt). Kein Pre-Warming.
- **HMAC-SHA256 signierte URLs** verhindern Disk-Filling-Angriffe.
- **Apache-direkt-Auslieferung** auf Cache-Hits via `.htaccess`-Rewrite. PHP läuft nur beim ersten Request einer Variante.
- **Custom `ColorProfile` Glide-Manipulator** normalisiert Imagick-Colorspace auf sRGB (fixt Display P3 / Adobe RGB Washout).
- **LQIP** (Low-Quality Image Placeholder) als Inline-Base64-JPEG (32 px, Blur 40, Q 40) als `background-image` — JS-frei.
- **Blurhash**-Generierung über `kornrunner/blurhash`, gecached in `_meta/`-Sidecars. Abrufbar via `Image::blurhash($src)` oder als opt-in `data-blurhash` Attribut.
- **Focal-Point**-Unterstützung über das optionale [`focuspoint`](https://github.com/yakamara/redaxo_focuspoint) Addon (`med_focuspoint` Feld). Per-Call-Override über `focal:` Argument.
- **Preload** für Above-the-fold-Bilder via `<link rel="preload" as="image">` Injection in den `<head>` über `OUTPUT_FILTER`.
- **SVG/GIF Pass-through** — keine Transformation, schlichtes `<img>` mit intrinsischen Maßen.
- **CDN-Override** (ImageKit / Cloudinary / Imgix-kompatibel) via `cdn_base` + `cdn_url_template` mit `{w}/{q}/{fm}/{src}` Tokens.
- **Hybrid Public API**: statisches `Image::picture(...)` als One-Liner für den Standardfall + fluentes `Image::for($src)->...->render()` für komplexere Fälle. Analoges Design für `Video`. `Pic` als Statamic-style Kurzalias.
- **`REX_PIC[src="..." alt="..." …]` Placeholder** für Inhaltspflege in Textfeldern / WYSIWYG, parsed über `OUTPUT_FILTER` — gleiche Pipeline wie die PHP-API.
- **Backend-Settings-Seite** mit vier Tabs: Allgemein, Placeholder, CDN, Sicherheit & Cache.
- **Dokumentations-Tab** im Backend rendert die `README.md` direkt (`subPath: README.md`).
- **`CACHE_DELETED` Hook** leert den Addon-Cache automatisch beim REDAXO-Cache-Reset.
- **Backend-Cache-Reset-Button** auf der Sicherheit-Seite für gezieltes Leeren ohne kompletten REDAXO-Cache-Clear.
- **Sign-Key-Regenerate-Button** für gezielte Invalidierung aller bisher signierten URLs.

### Pipeline architecture

- `Pipeline/ImageResolver` (rex_media → ResolvedImage)
- `Pipeline/MetadataReader` (intrinsische Maße + Blurhash + Focal, gecached in `meta.json` Sidecars per Asset-mtime)
- `Pipeline/ResolvedImage` (readonly Value Object)
- `Pipeline/SrcsetBuilder` (next/image dual-pool: `device_sizes` + `image_sizes`, capped at min(intrinsisch, width))
- `Pipeline/UrlBuilder` (signierter Glide-URL oder CDN-URL)
- `Pipeline/Placeholder` (Glide-generated Base64 LQIP)
- `Pipeline/Preloader` (Static-Queue, gedrained vom OUTPUT_FILTER)
- `View/PictureRenderer` + `View/PassthroughRenderer`
- `Glide/{Server,Endpoint,Signature,ColorProfile}`
- `Parser/REXPicParser`

### Requirements

- REDAXO 5.13+
- PHP 8.2+
- Imagick (empfohlen, nötig für AVIF-Output)
- Optional: `focuspoint` Addon für visuelle Focal-Point-Pflege

### Notes
